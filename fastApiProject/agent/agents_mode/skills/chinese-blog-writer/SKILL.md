# 中文技术博客生成器

## 概述

根据用户提供的主题和技术要点，生成一篇风格统一、结构规范的中文技术博客文章。输出格式为 Markdown，适合直接发布到博客平台或公众号。

## 使用场景

- 学完一个新技术概念，想写一篇教程
- 做完一个技术 demo，想分享经验
- 需要把英文技术文档转化为中文博客
- 团队内部技术分享的文稿

## 写作规范

### 文章结构（必须遵循）

每篇文章必须包含以下结构：

```markdown
# 标题（简洁有力，15字以内）

> 一句话摘要：用一句话说清楚这篇文章讲什么

## 引言
- 用一个场景或问题引入（为什么读者应该关心这个话题）
- 说明这篇文章会覆盖什么内容
- 预期读者：前端/后端/全栈/初学者等

## 核心概念
- 用通俗的语言解释核心概念
- 必须包含至少一个类比（让非专业读者也能理解）
- 如果有多个概念，用对比表格呈现

## 实战演示
- 完整可运行的代码示例
- 逐步拆解，每一步都要解释"为什么这样做"
- 关键代码行加注释

## 常见问题 / 踩坑记录
- 至少列出 2-3 个常见问题
- 每个问题给出解决方案

## 总结
- 3-5 个核心要点的 bullet list
- 下一步学习建议或延伸阅读
```

### 写作风格

| 要求 | 说明 |
|------|------|
| 语气 | 对话式，像在和朋友聊天，不要学术腔 |
| 人称 | 第一人称"我"为主，拉近距离 |
| 段落 | 每段不超过 4 行，保持阅读节奏 |
| 专业术语 | 首次出现时附英文原文，如：渐进式披露（Progressive Disclosure）|
| 类比 | 每个核心概念至少一个生活化类比 |
| 过渡 | 段落之间要有自然过渡，不要突兀跳转 |

### 代码规范

- 代码块必须标注语言类型（```python, ```javascript 等）
- 关键行加中文注释
- 完整可运行，不要省略 import 语句
- 如果代码超过 30 行，拆分成多个代码块，中间穿插解释

### 格式规范

- 标题层级：最多到 h3（###），不要用 h4 及以下
- 列表项：保持同一层级的格式统一
- 图片/图表：用 `![描述](url)` 格式，描述要写清楚
- 强调：**粗体**用于关键概念，`行内代码`用于技术术语

## 输出要求

1. 输出纯 Markdown 格式
2. 文章长度：1500-3000 字（不含代码）
3. 代码示例必须完整可运行
4. 文末不加"关注我"等推广内容

## 示例输入输出

**输入：**
> 主题：Python 装饰器
> 要点：什么是装饰器、语法糖原理、常用场景（计时器、权限校验、缓存）

**输出示例（节选）：**

```markdown
# Python 装饰器：给函数穿上"外套"

> 一句话摘要：装饰器就是一个函数，它接收一个函数，返回一个增强版的函数。

## 引言

你有没有遇到过这种情况？你写了 20 个 API 接口，每个接口都要做权限校验。
你把同样的 if-else 复制了 20 遍，改一个地方就要改 20 次。

这时候你需要的就是装饰器——它能把重复的逻辑抽出来，像穿外套一样"套"在函数上面。

这篇文章会讲清楚三件事：装饰器是什么、怎么写、用在哪。
```

## 质量检查清单

生成文章后，自查以下项目：

- [ ] 标题是否简洁有力（15 字以内）
- [ ] 是否有一句话摘要
- [ ] 引言是否用场景/问题引入
- [ ] 每个核心概念是否有类比
- [ ] 代码示例是否完整可运行
- [ ] 是否列出了常见问题
- [ ] 总结是否有要点 bullet list
- [ ] 段落长度是否合理（不超过 4 行）
- [ ] 专业术语是否附了英文原文
